# 要件定義書：少規模データによる効率的手話VLA学習フレームワーク

## 1. 研究の背景と課題（Background & Motivation）
近年、視覚・言語・行動を単一のネットワークで統合するVLA（Vision-Language-Action）モデルは、汎用ロボット制御のパラダイムとして大きな注目を集めています。しかし、現行のVLAモデルには以下の決定的な障壁が存在します。

- **データ飢餓問題（Data Hunger）**: 膨大な物理試行データ（Actionデータ）が必要であり、収集コストが極めて高い。
- **ドメイン適応の困難さ**: データが少ないドメイン（日本手話など）において性能が大きく低下する。
- **計算リソースの制限**: 企業レベルの超巨大計算サーバーがない大学の研究室や中小企業では、既存の巨大VLAを再学習させることが困難。

## 2. 研究の目的とゴール（Objectives & Goals）
本研究では、手話動作を具体的な検証ドメインとし、**「最小限の物理試行（Few-shot / Sample-efficient）で未知のアクションを学習・実行できる高効率なVLA学習フレームワーク」**の確立を目指します。

- **ゴール**: 各手話動作あたりわずか **3〜10回程度** の実演データ（MediaPipeで抽出した骨格座標系列）から、新しい手話の認識、そのセマンティクス（意味）の理解、およびロボットアバター・実機（Turtlesim/Gazeboなど）の制御軌道生成（Action）を爆速で追加学習・実行できる手法を構築する。

---

## 3. システム要件（System Requirements）

### 3.1 入力（Vision / Perception）
- **入力ソース**: Mac内蔵カメラなどのRGBカメラ映像。
- **特徴抽出**: [MediaPipe Hands](https://github.com/google-ai-edge/mediapipe) を使用。
  - 右手（21箇所 × 3次元座標 [x, y, z] = 63次元）
  - 左手（21箇所 × 3次元座標 [x, y, z] = 63次元）
  - 合計 126次元 の時系列浮動小数点数データ。

### 3.2 処理・モデル（Language / Model / PEFT）
- **特徴表現と意味解釈**:
  - 時系列座標データ（Vision特徴）を、言語的意味（Languageラベル）または制御コード（Actionベクトル）にマッピングする。
- **学習・適応アルゴリズム**:
  - 数個の学習データでも過学習を防ぐ軽量な時系列ネットワーク（LSTM、GRU、または空間・時間アテンション付きのTransformer）。
  - 必要に応じて、事前学習モデルに対するパラメータ効率的ファインチューニング（PEFT / LoRAなど）や、データ拡張（データオーグメンテーション）の手法を適用。

### 3.3 出力（Action / Control）
- **実行環境**: ROS2 (Humble / Pixiで管理された環境)
- **制御対象**: 
  - 第一段階: [Turtlesim](https://docs.ros.org/en/humble/Tutorials/Beginner-CLI-Tools/Introducing-Turtlesim/Introducing-Turtlesim.html)（2次元上の平面移動：前進・後退・回転）
  - 第二段階: [Gazebo](https://gazebosim.org)（3次元シミュレータ上でのマニピュレータ、または人型/多関節ロボットのアクション制御）
- **出力プロトコル**: ROS2 Topic（例: `/turtle1/cmd_vel` に `geometry_msgs/msg/Twist` をパブリッシュ）

## 4. 機能要件（Functional Requirements）

| ID | 機能名 | 詳細要件 |
|---|---|---|
| F-1 | 骨格トラッキング機能 | カメラ入力から、リアルタイムで両手の骨格（計42点）を検出・描画する。 |
| F-2 | データレコーディング機能 | 特定の手話動作を実演しながら、スペースキー操作等で開始・終了を切り替え、座標時系列をCSVに保存できること。 |
| F-3 | データ前処理機能 | カメラ距離や位置のばらつきを吸収するため、手首（リスト）を基準にした相対座標変換や正規化を行う。 |
| F-4 | 少数データ学習機能 | 3〜10個程度のCSVデータから、過学習を回避しつつ新たな手話クラスを識別・軌道へ変換するモデルを訓練できること。 |
| F-5 | リアルタイム推論機能 | 稼働中のカメラ映像から、学習済みモデルがリアルタイムで手話を識別・動作生成すること。 |
| F-6 | ROS2ロボット制御機能 | 推論された手話アクション（または意味ラベル）を、ROS2のノードを経由してトピック通信によりロボットに送信・駆動させること。 |

---

## 5. 非機能要件・制約条件（Non-Functional Requirements）

- **動作環境の制限**:
  - 開発環境は macOS (Mac教室のMacを含む) とする。
  - ROS2 は `pixi` (Conda-Forge/Robostack Humble) を用いて、システムに管理者権限（パスが通せない制約）がなくてもユーザー権限内で実行可能な設計とする。
- **パフォーマンス要件**:
  - カメラ入力からROS2への制御送信まで、人間が遅延を感じない遅延時間（100ms以内）でのリアルタイム処理を目標とする。
- **データの少なさ**:
  - 各ジェスチャーにつき、最小3回ずつのデモンストレーションで一定の認識およびアクション生成を実現する。

---

## 6. 評価指標（Evaluation Metrics）

研究テーマである「データ効率的学習」の有効性を証明するため、以下の指標で定量的・定性的な評価を行います。

1. **データ数（N-shot）に対する精度推移（Learning Curve）**:
   - 学習データ数（N = 1, 3, 5, 10, 30）を変化させた際の、手話認識の正解率（Accuracy）および制御再現度の推移。
2. **計算効率（Training Resource）**:
   - ファインチューニング時の学習所要時間およびメモリ消費量（PEFT導入前後での比較）。
3. **対ノイズ堅牢性**:
   - カメラの画角や撮影距離が異なる環境で、どの程度正しく認識できるか。
